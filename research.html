Scientific codebase RAG

Current work / University of Utah / Retrieval-augmented generation for scientific computing libraries

The current research target is RAG for scientific codebases, using KernelPack, a Python library for RBF-FD numerical methods, as the primary testbed. Scientific code retrieval has a failure mode that generic code RAG does not handle well: the user may ask in mathematical language while the codebase answers in API names, implementation details, and cross-references.

Jordan frames this as a trimodal retrieval problem. A good system has to retrieve code, mathematical meaning, and related implementation context at the same time. A query like "4th-order Laplacian" is not just a text search problem; it may require connecting numerical vocabulary to a class, a factory method, an operator name, and the function that assembles the solver.

Early experiments compare chunking strategies, code-aware embeddings, BM25 plus dense retrieval, reciprocal rank fusion, and LLM-generated source summaries. The current finding is useful because it is specific: after summary enrichment, expected chunks often appear in the top-10 neighborhood, but not reliably in the top-3. That makes ranking, not raw recall, the next bottleneck.

Next steps include a Qdrant migration, richer metadata for math terms and cross-references, hierarchical "retrieve small, read big" context expansion, and cross-encoder reranking.

Two RAG directions

The shared retrieval foundation supports two interfaces. The first is a coding-agent RAG layer that constrains AI assistants to retrieved source context so they do not invent functions, classes, or arguments. The second is a citizen-science RAG interface that translates plain-language simulation goals into grounded starter code for users who do not already know the solver vocabulary.

These projects diverge at the generation and interface layers, but they share the same evaluation premise: a system should not be trusted because it sounds fluent. It should be trusted only when retrieval, grounding, and execution behavior can be measured.

MSD capstone: insider threat detection

Machine learning capstone / CERT Insider Threat Dataset / behavioral activity logs

Jordan's capstone work built a multi-detector ensemble for identifying malicious insider behavior in enterprise activity logs. The system transforms logon, device, file, HTTP, email, and LDAP data into daily user features, then evaluates rolling 14-day windows with four independent detection strategies.

The ensemble combines an XGBoost active-exfiltration detector, an XGBoost forecast detector for pre-exfiltration warning, deterministic rules for high-confidence event chains, and an Isolation Forest anomaly baseline normalized by personal and peer-group behavior. A logistic ensemble fuses those signals so noisy anomaly alerts support, rather than dominate, escalation.

On the held-out Scenario 1 test split, the active detector reached 1.000 ROC-AUC and 0.970 average precision at the window level, with zero false positives at the default threshold. The forecast model caught 2 of 3 test-set insiders before exfiltration began, with 1 to 9 days of lead time. The ensemble comparison also showed why evaluation matters: letting anomaly scores dominate collapsed performance, while the weighted ensemble preserved precision.
